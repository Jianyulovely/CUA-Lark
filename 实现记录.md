## 感知层 — TreeParser（`perception/tree_parser.py`）

**作用：** 通过 pywinauto UIA backend 实时查询飞书界面 UI 树，返回元素坐标。

### 核心数据结构

```python
@dataclass
class UITreeSelector:
    control_type: str              # 如 "Button", "TabItem", "Text"
    name: str | None = None        # 精确匹配元素名称
    name_contains: str | None = None  # 模糊匹配

@dataclass
class UIElement:
    control_type: str
    name: str
    center_x: int
    center_y: int
    rect: tuple[int, int, int, int]
    _raw: Any   # pywinauto 原始元素
```

### 关键方法

| 方法 | 说明 |
|------|------|
| `connect(title_pattern)` | 连接飞书主窗口 |
| `connect_window(title, timeout)` | 连接指定标题的顶层窗口（用于弹窗） |
| `wait_for_new_window(known_titles, timeout)` | 等待新窗口出现 |
| `find(selector)` | 返回第一个匹配元素，失败返回 None |
| `find_all(selector)` | 返回所有匹配元素列表 |
| `get_all(visible_only=True)` | 返回当前窗口所有有效元素 |
| `scroll_and_find(selector, scroll_target)` | 滚动并查找元素 |

### 飞书 Electron 特性

- **`auto_id` 全为空**，只能用 `control_type + name` 匹配
- **输入框不暴露 `Edit` 控件**，通过占位文本 `Text` 节点定位
- **坐标是实时屏幕绝对坐标**，每次执行前必须重新查询
- **"创建日程"弹窗**是独立顶层窗口，必须用 `connect_window()` 单独连接

### connect() 实现说明

**问题：** 飞书窗口可能在后台/最小化；浏览器若有含"飞书"字样的标签页，正则匹配会命中多个窗口导致 `ElementAmbiguousError`。

**解决方案：** 使用 `win32gui.EnumWindows` 枚举所有窗口，精确匹配标题 `== "飞书"`（飞书主窗口标题恰好就是这两个字），找到后调用 `SW_RESTORE` + `SetForegroundWindow` 唤起并置前台。找不到则报错提示用户打开飞书。

```python
def connect(self, title: str = "飞书") -> None:
    handles = []
    def _enum(hwnd, _):
        if win32gui.GetWindowText(hwnd) == title:
            handles.append(hwnd)
    win32gui.EnumWindows(_enum, None)

    if not handles:
        raise RuntimeError(f"未找到标题为 '{title}' 的窗口，请先打开飞书桌面端")

    hwnd = max(handles, key=_area)
    win32gui.ShowWindow(hwnd, win32con.SW_RESTORE)
    win32gui.SetForegroundWindow(hwnd)
    time.sleep(0.8)
    self._app = Application(backend="uia").connect(handle=hwnd)
    self._win = self._app.top_window()
```

**关键决策：** 精确匹配 `"飞书"` 而非正则 `.*飞书.*`，避免浏览器标签页干扰。

## 执行层 — CUAExecutor（`execution/cua_executor.py`）

**作用：** 将逻辑动作映射为实际鼠标键盘操作，封装 pyautogui + pyperclip。

### 核心方法

| 方法 | 实现方式 | 说明 |
|------|----------|------|
| `click(x, y)` | `pyautogui.click()` | 鼠标单击绝对坐标 |
| `double_click(x, y)` | `pyautogui.doubleClick()` | 双击 |
| `type_text(text)` | `pyperclip.copy()` + `Ctrl+V` | 支持中文、emoji |
| `send_key(key_combo)` | `pyautogui.hotkey()` / `press()` | 快捷键，如 `"ctrl+k"`, `"enter"` |
| `scroll(x, y, direction, amount)` | `pyautogui.scroll()` | 滚轮 |
| `execute_chunk(chunk)` | 顺序调用 `_execute_one()` | 批量执行动作序列 |

### 中文输入方案

pyautogui 的 `typewrite()` 不支持非 ASCII 字符。解决方案：

```python
def type_text(self, text: str) -> None:
    pyperclip.copy(text)     # 写入剪贴板
    time.sleep(0.05)
    pyautogui.hotkey("ctrl", "v")  # 粘贴
```

**副作用：** 会短暂覆盖系统剪贴板内容。

## 视觉层 — UITARSClient（`core/api_client.py`）

**作用：** 截图 + 自然语言指令 → 结构化动作（坐标 + 动作类型）；以及视觉问答（验证界面状态）。

### 模型现状

| 模型 | 状态 | 说明 |
|------|------|------|
| `doubao-1-5-ui-tars-250428` | ❌ 待权限 | 专门为 GUI 微调，坐标精度最高 |
| `Doubao-Seed-1.6-vision`（`VISION_EP`）| ✅ 当前使用 | 通用视觉模型，prompt 工程替代 |

### predict()：动作预测

系统提示要求输出 `Thought: ...\nAction: ...` 格式，坐标 0-1000 相对坐标，附示例防止模型输出自然语言。

正则解析支持单双引号（兼容通用模型输出差异）：

```python
r'click\(start_box=["\']?\[\[(\d+),\s*(\d+),\s*(\d+),\s*(\d+)\]\]["\']?\)'
```

坐标换算：`0-1000 相对坐标 → 屏幕绝对像素`

```python
def _scale(v1, v2, screen_dim):
    center = (v1 + v2) / 2
    return int(center / 1000 * screen_dim)
```

### verify()：视觉验证（新增）

与 `predict()` 使用不同的 system prompt——问答模式，要求模型回答 YES/NO 并给出中文理由：

```
Result: YES
Reason: 消息"xxx"已清晰出现在聊天记录底部
```

返回 `VerifyResult(success=True/False, message="...")`，供 `AgentLoop.run_with_verify()` 判断是否需要重试。

## 规划层 — 模板库（`planning/template_library.py`）

**作用：** 将槽位（用户意图参数）展开为具体执行步骤序列。

### TemplateStep 数据结构

```python
@dataclass
class TemplateStep:
    step_id: str
    routing: Literal["tree", "keyboard", "vision"]
    description: str
    selector: UITreeSelector | None = None   # tree 步骤：标准选择器
    find_fn: Any | None = None               # tree 步骤：自定义查找函数
    action_type: Literal["click", "type", "key"] = "click"
    text: str | None = None
    key_combo: str | None = None
    wait_ms: int = 500
    switch_to_window: str | None = None      # 执行后切换到新顶层窗口
```

### 已实现模板

模板函数现在返回 `TaskTemplate`（步骤列表 + 验证函数），而非原来的 `list[TemplateStep]`。

#### 模板一：`send_message`（发送消息）

| 步骤 | 路由 | 说明 |
|------|------|------|
| s1 | keyboard | `Ctrl+K` 打开搜索，光标自动聚焦输入框 |
| ~~s2~~ | ~~tree~~ | 已删除：搜索框在 Chromium 区不可见 |
| s3 | keyboard | 输入联系人名称 |
| s4 | tree | 点击"联系人"筛选按钮 |
| s5 | **vision** | 视觉识别并点击搜索结果中的联系人 |
| s6 | tree | 点击消息输入框（占位文本 `'发送给 {name}'`） |
| s7 | keyboard | 输入消息内容 |
| s8 | keyboard | `Enter` 发送 |

#### 模板二：`create_event`（创建日程）

| 步骤 | 路由 | 说明 |
|------|------|------|
| e1 | tree | 点击"日历" TabItem |
| e2 | tree | 点击"创建日程"按钮，`switch_to_window="创建日程"` |
| e3 | **vision** | 点击标题输入框（空框，UI 树不可见） |
| e3b | keyboard | 输入标题 |
| e4 | tree（find_fn）| 点击开始时间框（HH:MM 格式，x<900，第0个）|
| e4b-d | keyboard | `Ctrl+A` → 输入时间 → `Tab` 确认 |
| e5 | tree（find_fn）| 点击结束时间框（第1个）|
| e5b-d | keyboard | 同上 |
| e6 | tree | 点击"保存"按钮 |

**关键发现：** 时间框支持键盘直接输入，无需操作下拉列表。

**时间框识别方式（`_find_nth_time_text`）：**
- 过滤 `Text` 控件，名称匹配 `\d{1,2}:\d{2}`
- 排除 x > 900 的日历刻度（x ≈ 1106）
- 按 x 升序：第0个 = 开始时间，第1个 = 结束时间

#### 模板三：`add_todo`（创建待办文档）

| 步骤 | 路由 | 说明 |
|------|------|------|
| d1 | tree | 点击"云文档" TabItem |
| d2 | tree | 点击"创建"按钮，展开下拉菜单 |
| d3 | tree | 点击 `Button name="TitleBarMenu-CREATE_DOC"`，等待编辑器加载（wait 2500ms）|
| d4 | **vision** | 点击文档标题输入区域（空框，UI 树不可见）|
| d4b | keyboard | 输入文档标题 |
| d5 | keyboard | Enter 将光标移入正文区域 |
| d6 | keyboard | 输入 `/` 打开命令菜单 |
| d7 | keyboard | 输入 `待办` 过滤命令菜单 |
| d8 | keyboard | Enter 选中"待办事项"块 |
| d9_i | keyboard | 逐条输入待办事项文本 |
| d10_i | keyboard | Enter 换行到下一条（最后一条不加）|

**槽位：** `title`（文档标题）、`items`（多条待办，顿号/逗号分隔）

**步骤生成：** `items` 字段按 `[、,，]` 正则分割，循环生成 d9_i / d10_i 步骤，支持任意数量。

**关键发现：**
- 新建文档按钮 UI 树名称为 `TitleBarMenu-CREATE_DOC`（内部 ID），非"新建文档"可读标签
- 文档编辑器在飞书主窗口内打开，无需 `switch_to_window`（与日历弹窗不同）
- 标题输入框为空时 UI 树不可见，使用视觉模型定位
- **`/` 命令必须用 `send_key("/")` 按键触发**，用 `type_text("/")` 走 Ctrl+V 粘贴无法唤出命令菜单
- `/` 命令菜单支持中文关键字过滤，`待办` → Enter 可选中"待办事项"块
- 连续 Enter 可在待办块内创建多条条目
- 验证不检查编辑器正文（文档创建后仍处于"保存中..."状态，正文区不可见），改为检查左侧文档库列表中是否出现目标标题，验证前等待 2 秒使保存状态稳定



**作用：** 定义验证相关数据结构，提供截图工具供验证函数调用。

### 数据结构

```python
@dataclass
class VerifyResult:
    success: bool
    message: str    # 成功/失败说明

@dataclass
class TaskTemplate:
    intent: str
    steps: list[TemplateStep]
    verify_fn: Callable[[TreeParser, UITARSClient], VerifyResult]
```

### 验证策略

两个模板均使用**视觉验证**：执行完成后截图，调用 `UITARSClient.verify()` 以问答方式确认业务结果：

| 模板 | 验证问题 |
|------|---------|
| `send_message` | `"消息'{content}'是否已出现在聊天记录里？"` |
| `create_event` | `"日历视图中是否出现了标题为'{title}'的日程？"` |

**设计决策：** 选择视觉验证而非 UI 树验证，原因是聊天记录和日历内容均在 Chromium 渲染区，UI 树不可见；视觉验证能真正确认业务结果，而非仅确认控件状态变化。

## 执行循环 — AgentLoop（`execution/agent_loop.py`）

**作用：** 顺序执行 TemplateStep 列表，串联 TreeParser、CUAExecutor、UITARSClient；执行完成后调用验证并自动重试。

### 路由分发逻辑

```
tree     → find_fn(parser) 或 parser.find(selector) → executor.click(x, y)
keyboard → executor.type_text(text) 或 executor.send_key(key_combo)
vision   → 截图(base64) → ui_tars.predict(instruction) → executor.click(x, y)
```

### run_with_verify()：带验证的执行（新增）

```python
def run_with_verify(template: TaskTemplate) -> None:
    for attempt in 1..max_retry+1:
        run(template.steps)          # 执行所有步骤
        result = template.verify_fn  # 视觉验证
        if result.success: return
        log.warning(f"验证失败（第{attempt}次）: {result.message}")
        if attempt < max_attempts: continue
        raise RuntimeError("多次重试后仍验证失败")
```

重试时自动重新连接飞书主窗口（`parser.connect()`），避免窗口状态残留影响下次执行。

重试次数由 `config.max_retry`（默认 2）控制，共最多执行 3 次。

### 窗口切换（switch_to_window）

步骤执行完成后，如果设置了 `switch_to_window`，调用 `parser.connect_window()` 等待并连接新窗口（超时 6 秒）。用于"创建日程"弹窗场景。

### 截图方案

```python
screenshot = pyautogui.screenshot()
buffer = BytesIO()
screenshot.save(buffer, format="PNG")
return base64.b64encode(buffer.getvalue()).decode("utf-8")
```

## 飞书 UI 树实测发现

| 场景 | 发现 | 解决方案 |
|------|------|----------|
| 搜索框（Ctrl+K 后）| 在 Chromium 渲染区，UI 树不可见 | Ctrl+K 后光标自动聚焦，直接 type |
| 搜索结果列表 | 在 Chromium 渲染区，UI 树不可见 | 视觉模型识别点击 |
| 消息输入框 | 不暴露 Edit 控件，但占位文本可见 | 查找 `Text name_contains="发送给"` |
| "联系人"筛选按钮 | UI 树可见，Ctrl+K 后出现 | `Button name="联系人"` |
| "创建日程"弹窗 | 独立顶层窗口（非 modal child）| `connect_window("创建日程")` |
| 日程标题输入框 | 空框，无占位文本，UI 树不可见 | 视觉模型识别点击 |
| 日程时间框 | UI 树可见，格式 `HH:MM` | `find_fn` 按坐标区分开始/结束 |
| 日程时间输入 | 支持键盘直接输入 | `Ctrl+A` → type → `Tab` |
| 日历视图时间刻度 | 与时间框同名（如 `'13:00'`），x ≈ 1106 | 过滤 `center_x > 900` |
| 飞书主窗口标题 | 精确为 `"飞书"` 两字 | `win32gui.GetWindowText(hwnd) == "飞书"` 精确匹配，避免浏览器标签页干扰 |

## 对话层 — DialogueManager（`dialogue/dialogue_manager.py`）

**作用：** 将用户自然语言输入转化为结构化的意图 + 槽位，不足时最多追问 2 轮。

### DialogueState

```python
@dataclass
class DialogueState:
    intent: str = ""            # "send_message" / "create_event" / "unknown"
    filled_slots: dict = ...
    missing_slots: list = ...
    turn_count: int = 0         # 已追问次数，上限 config.max_clarification_turns=2
```

### LLM 调用设计

一次 API 调用同时返回意图、槽位、缺失项、追问语句（四合一），避免多次往返：

```json
{
  "intent": "send_message",
  "slots": {"recipient": "张三", "content": "今天开会取消了"},
  "missing": [],
  "question": ""
}
```

**System Prompt 关键规则：**
- 时间自然语言统一转换为 `HH:MM`（"下午三点" → "15:00"）
- 识别不到已知意图时 `intent` 填 `"unknown"`
- 只输出 JSON，不输出其他文字

### process() 返回语义

| 返回值 | 含义 |
|--------|------|
| `(intent, slots)` | 槽位完整，可以执行 |
| `None` | 已打印追问语句，等待用户下一轮输入 |
| 抛出 `RuntimeError` | 追问超限或意图未知，需要重新描述 |

### 槽位定义

| 意图 | 必填槽位 |
|------|---------|
| `send_message` | `recipient`, `content` |
| `create_event` | `title`, `start_time` |
| `create_event`（选填）| `end_time`（默认 start_time + 1 小时）|
| `add_todo` | `title`, `items` |

## 意图路由 — IntentRouter（`dialogue/intent_router.py`）

**作用：** 意图名 → 模板函数的注册表，解耦对话层和规划层。

```python
INTENT_REGISTRY = {
    "send_message": send_message_steps,
    "create_event": create_event_steps,
    "add_todo":     add_todo_steps,
}

def route(intent: str, slots: dict) -> TaskTemplate:
    return INTENT_REGISTRY[intent](slots)
```

新增模板时只需在 `INTENT_REGISTRY` 里加一行，不改其他代码。

## 主入口 — main.py（`cua_lark/main.py`）

**作用：** Terminal 对话循环，连接飞书后持续接受自然语言指令。

**运行方式：**
```bash
python cua_lark/main.py
# 或
python -m cua_lark.main
```

**流程：**
1. 连接飞书主窗口（自动唤起）
2. 进入 REPL 循环：用户输入 → DialogueManager → IntentRouter → AgentLoop
3. 每次任务完成后重置 DialogueManager 状态，准备接受下一条指令
4. 输入 `quit` / `退出` 退出

**sys.path 修复：** 文件顶部加 `sys.path.insert(0, project_root)`，使 `python cua_lark/main.py` 直接运行时也能正确导入 `cua_lark` 包。
